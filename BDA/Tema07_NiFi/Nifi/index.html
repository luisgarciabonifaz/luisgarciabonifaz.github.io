    7.1. Apache NiFi

    Apache NiFi es una herramienta de integración y automatización de flujos de datos que permite el diseño, la gestión y la monitorización de flujos de datos de una manera visual y sencilla. Su propósito es mover y transformar datos entre diferentes sistemas de forma eficiente y confiable.

    7. Conceptos clave

    Flujo de datos (Data Flow):

    Un flujo de datos en NiFi es una representación visual de cómo los datos se mueven de una fuente a un destino. Cada flujo puede incluir pasos de procesamiento, filtrado, transformación y distribución de datos.

    Processor:

    Es el componente básico de NiFi que realiza una operación sobre los datos. Un Processor puede ser de muchos tipos, por ejemplo:

    • Leer archivos de un servidor FTP.
    • Escribir datos en una base de datos.
    • Transformar formatos de archivo (JSON, XML, CSV).

    FlowFile:

    Un FlowFile representa un mensaje o dato individual que se mueve a través del sistema NiFi. Cada FlowFile tiene un contenido (los datos) y atributos (metadatos como nombre de archivo, origen, etc.).

    Controladores de servicios (Controller Services):

    Son servicios compartidos que proporcionan acceso a recursos comunes, como bases de datos o servidores de mensajes.

    Data Provenance (Trazabilidad de Datos):

    NiFi incluye una función llamada provenance, que permite rastrear cada FlowFile desde su origen hasta su destino. Esto es muy útil para auditar qué ha ocurrido con los datos.

    7.2. Arquitectura de Apache NiFi

    La arquitectura de NiFi se centra en tres componentes principales:

    • NiFi Web UI (Interfaz gráfica):
      • Permite a los usuarios diseñar, configurar y monitorizar los flujos de datos de manera visual.
      • Los flujos se crean arrastrando y soltando Processors y conectándolos entre sí.
      • Se accede a través de un navegador web, ofreciendo una experiencia de usuario sencilla e intuitiva.
    • Motor de procesamiento de flujo (Flow Engine):
      • Responsable de ejecutar los flujos de datos.
      • Puede procesar los datos en tiempo real o en modo batch.
      • Administra la ejecución de Processors, la encolación de FlowFiles y la distribución de datos entre Processors.
    • Clustering (Agrupación):
      • Apache NiFi soporta el clustering, lo que permite distribuir la carga de procesamiento entre múltiples nodos. Esto asegura alta disponibilidad y escalabilidad.

    7.3. Características clave

    Arrastrar y soltar (Drag and Drop): La interfaz gráfica permite que los usuarios diseñen flujos sin necesidad de escribir código, solo arrastrando componentes.

    Flujo en tiempo real y en batch: NiFi es capaz de procesar datos en tiempo real, pero también soporta procesos batch. Se adapta tanto a la ingesta continua de datos como a tareas programadas.

    Enrutamiento y transformación de datos: Con los Processors, NiFi permite enrutamiento condicional, transformación y filtrado de datos según su contenido o atributos.

    Alta disponibilidad y tolerancia a fallos: Mediante la agrupación en clústeres, NiFi distribuye el procesamiento entre nodos, garantizando que si un nodo falla, los otros continúan procesando los datos.

    Seguridad: Autenticación y autorización basada en roles. Encriptación de datos en tránsito y en reposo. Control detallado de acceso para proteger los flujos de datos.

    Monitorización y trazabilidad: NiFi permite a los administradores monitorear flujos en tiempo real y rastrear el recorrido de cada dato dentro del sistema.

    7.4. Aplicaciones comunes de Apache NiFi

    • Integración de datos: Conectar distintas fuentes de datos heterogéneas, como bases de datos, APIs, sistemas de archivos, servidores FTP, etc.
    • ETL (Extract, Transform, Load): NiFi permite extraer datos de diferentes fuentes, transformarlos según sea necesario (por ejemplo, convertir formatos) y cargarlos en un sistema de destino (bases de datos, almacenes de datos, etc.).
    • Streaming de datos: Procesar flujos de datos en tiempo real, como por ejemplo, ingesta de datos de sensores IoT.
    • Integración con sistemas en la nube: Transferir y transformar datos hacia o desde sistemas de almacenamiento en la nube como Amazon S3, Google Cloud Storage, entre otros.

    7.5. Procesadores Comunes

    • Entrada/Salida
      • GetFile: Lee archivos de sistema local
      • PutFile: Escribe archivos en sistema local
      • GetHTTP: Obtiene datos via HTTP
      • PublishKafka: Envía datos a Apache Kafka
    • Transformación
      • ConvertRecord: Convierte entre formatos
      • ExecuteScript: Ejecuta scripts personalizados
      • SplitText: Divide archivos de texto
      • MergeContent: Combina múltiples FlowFiles
    • Enrutamiento
      • RouteOnAttribute: Enruta basado en atributos
      • ControlRate: Controla la velocidad del flujo
      • UpdateAttribute: Modifica atributos

    7.6. Casos de uso

    7.6.1. Diferencias entre FetchFile y GetFIle

    GetFile:

    • Lee archivos de un directorio local del sistema
    • Monitorea continuamente el directorio en busca de nuevos archivos
    • Puede eliminar o mover el archivo original después de leerlo
    • Ideal para vigilar carpetas donde se depositan archivos nuevos

    FetchFile:

    • Lee archivos basándose en una ruta específica proporcionada en los atributos del flowfile
    • No monitorea directorios - necesita que otro procesador le indique qué archivo leer
    • Normalmente se usa en conjunto con ListFile o GetFile
    • Mejor para lecturas selectivas o bajo demanda

    • Property Name: route

    • clone: ${filename:equals('true')}

    7.6.3. Escritra en BD

    Para guardar datos en BD necesitamos un procesador PutDatabaseRecord y un controlador de servicio DBCPConnectionPool con estas propiedades:

    DBCPConnectionPool:

    • Database Connection URL: jdbc:mysql://localhost:3306/mibasedatos
    • Database Driver: mysql-connector-java.jar
    • Database Driver Location: /path/to/mysql-connector.jar

    PutDatabaseRecord:

    • Database Connection Pooling Service: DBCPConnectionPool
    • Table Name: Nombre_Tabla
    • Record Query Mode: INSERT
    Ayuda

    Si tienes problemas de codificación de caracteres con los origenes de datos puede que tengas que adaptar las tablas al conjunto de caracteres adecuado. En MySQL debes usar este comando:

    ALTER TABLE tu_tabla CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

    7.6.5. Notify y Wait

    Para sincronizar varias ramas de un flujo de datos puedes utiizar los procesadores Wait y Notify. Estos serian los pasos:

    1. Añadir un procesasor Notity al final de cada rama que queramos sincronicar

      • Asigna el mismo nombre a todos los Release Signal Identifier de los Notify

    2. Añadir un procesadsor Wait al inicio de la unificaicón de las ramas

      • Asigna el mismo nombre que usaste en los procesasores Notify al Release Signal Identifier del Wait
      • Asinga al Target Signal Count del Wait el numero de Notify que tengas para permitir la sincronización cuando todos acaben

    3. Configurar un Distributed Map Cache Service

      • En la pestaña Controller Services, haz clic en + para añadir un nuevo servicio.
      • Busca y selecciona DistributedMapCacheServer. Este será el servicio que actuará como caché.
      • Configura el servicio:
        • Asigna un puerto en el parámetro Port (por ejemplo, 4557 o cualquier puerto libre en tu sistema).

    4. Configurar un Distributed Map Cache Client Service

      • En la misma sección de Controller Services, añade otro servicio, esta vez selecciona DistributedMapCacheClientService.
      • Configura el cliente para que apunte al servidor configurado en el paso anterior:
      • En el parámetro Distributed Map Cache Server Hostname, ingresa localhost (o el nombre del host del servidor NiFi si es remoto).
      • En el parámetro Port, ingresa el mismo puerto que configuraste en el DistributedMapCacheServer (por ejemplo, 4557).

    5. Asociar el mismo Distributed Map Cache Client al Notify

      • Vuelve al procesador Notify.
      • En el campo Distributed Cache Service, selecciona el DistributedMapCacheClientService que configuraste en el paso anterior.

    6. Asociar el mismo Distributed Map Cache Client al Wait

      • Vuelve al procesador Wait.
      • En el campo Distributed Cache Service, selecciona el mismo DistributedMapCacheClientService.

    7.7. API REST de NiFi

    Apache NiFi ofrece una API REST que permite interactuar de forma similar al interfaz gráfico. Esta API es útil para administrar flujos de datos, configurar procesadores, obtener métricas y realizar tareas administrativas.

    La API REST de NiFi suele estar disponible en la siguiente ruta:

    http://<host>:<puerto>/nifi-api

    Las peticiones siguen al nomenclatura /nifi-api/, de manera que con nuestra máquina virtual podríamos acceder a información sobre la instalación mediante

     https://localhost:8443/nifi-api/flow/about.

    Se puede consultar todas los servicios disponibles en la documentación oficial

    Algunos de los servicios más destacados son:

    • /access: para autenticarnos y obtener el token de acceso a Nifi
    • /controller: permite obtener la configuración de los controladores, gestionar el cluster y crear tareas para informes.
    • /controller-services: permite gestionar los servicios de controlador y modificar sus referencias.
    • /flow: para obtener metadatos de los flujos de datos, estado de los componentes y el histórico de consultas.
    • /process-groups: Para cargar e iniciar plantillas y crear componentes.
    • /processors: Permite crear y planificar un procesador y definir sus propiedades.
    • /connections: Para crear conexiones, definir las prioridades de las colas y modificar el destino de las conexiones.
    • /flowfile-queues: Permite ver el contenido de loas colas, descargar sus flowfiles y vaciar las colas.

    Para iniciar un grupo de procesasores con esta API REST deberiamos hacer los siguiente:

    1. Obtener el tocken para la API
    curl -k -X POST \
-d "username=nifi&password=nifinifinifi" \
https://localhost:8443/nifi-api/access/token
    1. Obtener el ID del grupo de procesadores

    1. Lanzar el grupo de procesadores 

      curl -k -X PUT   -H "Content-Type: application/json"   -H "Authorization: Bearer <token>"   -d '{
    "id": "<ProceesGroupID>",
    "state": "RUNNING"
  }'   https://localhost:8443/nifi-api/flow/process-groups/<ProceesGroupID>

    2. Parar el grupo de procesadores 

      curl -k -X PUT   -H "Content-Type: application/json"   -H "Authorization: Bearer <token>"   -d '{
    "id": "<ProceesGroupID>",
    "state": "STOPPED"
  }'   https://localhost:8443/nifi-api/flow/process-groups/<ProceesGroupID>

    7.8. Actividades

    7.8.1. Actividad 1 . Encuentas

    La jefatura de estudios de un instituto quiere crear un cuadro de mando con datos de la satisfacción de los alumnos y los profesores con distintos aspectos del funcionamiento y las instalaciones del centro.

    El cuadro de mando debe tener algunos filtros como:

    • Los departamentos a los que pertenecen los profesores
    • Familia profesonal de los estudios del alumno
    • Los estudios que realizan los alumnos

    Para ello nos facilita los datos extraidos de los las encuestas gestionadas a traves de una plataforma de e-learning. Los datos estan en formato csv.

    Estos son los ficheros que nos facilitan:

    • Encuentas
    • Completadas
    • Preguntas
    • Resultados

    Estudio de los datos:

    • Data Profiling
    • Limpieza de datos

    Almacenamiento

    • Plantea una estructura de almacenamiento de los datos limpios y preparados para hacer el analisis y crear el cuadro de mando.

    • Data Lake o Data Warehouse o Ambos

    • ....

WHERE Items.descripcion LIKE 'FAMILIA%';
